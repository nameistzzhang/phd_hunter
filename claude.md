# Claude Code 项目工作流程指令

本项目要求 Claude Code 在开发过程中严格遵守以下流程和规范。所有与用户的交互均使用**中文**。

## 1. 开发前：需求讨论与技术路线确认

- 在实现**任何一个功能**之前，必须先与用户进行充分的讨论，明确：
  - 功能的目标和输入输出
  - 涉及的技术栈、第三方库、API（如 CS Ranking、arXiv、Semantic Scholar）
  - 潜在难点和备选方案
- **禁止**在未得到用户明确同意的情况下直接开始编写实现代码。讨论结束后，等待用户确认技术路线后方可进入开发阶段。

## 2. 开发中：规划与分步实现

- 在开始编码前，先制定清晰的 **plan**，将任务拆解为可独立验证的小步骤。
- 按计划逐步实现，每完成一个步骤应进行自测。
- **必要情况下启动 subagent**：
  - 当代码量较大（例如单文件超过 300 行）或逻辑复杂时，启动一个 subagent 负责代码重构（refine）和语法检查。
  - Subagent 应输出经过 linting（如 pylint / eslint）且符合项目风格规范的代码。
  - 主 agent 负责整合 subagent 的产出并继续开发。

## 3. 开发后：自动停止并报告

- 完成一个完整功能（例如完成一个模块、一个 API 接入、一个解析器）后，**必须立即停止开发**，不得自动继续下一个功能。
- 然后向用户提交一份**开发报告**，内容包括：
  - 编写了哪些文件（相对路径）
  - 每个文件中实现的具体功能/函数
  - 代码的测试结果（手动测试或自动测试的结论，如“爬取 10 条数据成功”“摘要生成无报错”）
  - 尚未解决的问题或已知风险（若有）
- 等待用户的下一步指示。

## 4. 优化与更改：用户指导

- 用户会针对报告中的内容提出优化建议或要求修改。
- Claude Code 应逐一响应用户的指导，进行必要的代码调整、重构或补充测试。
- 每次优化后，简要说明修改了哪些地方以及新的测试结果。

## 5. 提交代码：达成共识后 Git Commit

- 只有**用户明确表示“可以提交”或“commit”**后，才执行 `git commit`。
- 提交前：
  - 确保所有相关文件已通过 `git add` 正确暂存。
  - 编写清晰、符合 Conventional Commits 规范的提交信息（中文或英文均可，但推荐英文）。
  - 示例：`feat: add arXiv paper fetching for each CS professor`
- 提交后向用户报告 commit hash 和变更摘要。

## 6. 交流语言

- **全程使用中文**与用户对话，包括讨论、报告、注释（代码注释除外，可保留英文）。
- 技术术语可以保留英文（如 “subagent”, “commit”, “API”），但前后应有中文说明。

## 7. 项目上下文（针对爬取 CS Ranking + 论文摘要 + 套磁方向）

本项目的最终目标是：
1. 爬取 CS Ranking 中的教授信息。
2. 对每位教授，获取其最新发表的文章（从 arXiv 或 Semantic Scholar）。
3. 调用 LLM 对文章进行摘要总结。
4. 根据用户的 research interest，自动生成个性化的套磁（cold email）方向建议。
5. 将所有结果结构化输出（如 JSON 或表格）。

请严格按照上述工作流程推进开发，确保每一步都经过用户确认。
